What is this?

A multi-agent AI research platform that generates Verkada-specific pre-sales discovery briefs. Give it a target account name, and it dispatches 20 specialized source clients in parallel, synthesizes findings through a persona-driven rule engine, and renders a sourced HTML brief ready for call prep.

The tool's value isn't "AI generates research" — generic LLMs already do that. The value is the persona file rule engine (persona/verkada-se.yml) that forces every output to be Verkada-specific and SE-grade. Every claim is sourced. Every discovery question traces to a fired trigger. Every section either has evidence or says insufficient_data.

Anti-genericness guarantee: Any sentence that could appear unchanged in a brief about a different company in the same industry is rejected during synthesis.

How it works

1

Source Collection

20 source clients run in parallel — SEC filings, news, job postings, NCES data, cooperative purchasing, leadership discovery, and more. Each writes structured JSON to sources/<company>/.

2

Subagent Analysis

Three subagents (company-bg, tech-and-pain, hiring-signals) each process relevant source data and produce structured findings with confidence levels.

3

Synthesis

Two synthesizers run in parallel: the main synthesizer (snapshot, pain hypotheses, displacement intel, champions) and the deals synthesizer (MEDDIC, GTM strategy, discovery questions). Both are grounded by the persona file.

4

Render

Brief JSON is rendered to a single-file HTML brief with source footnotes, confidence badges, interactive MEDDIC checklist, and an embedded chat panel for follow-up questions.

Model assignment: Source parsing uses Haiku. Subagent reasoning uses Sonnet. Final synthesis uses Opus. This is intentional — output quality comes from prompts and the persona file, not from model size at the wrong layer.

The Persona File

persona/verkada-se.yml is the platform's leverage file — a structured rule engine, not a description document. Every synthesis task loads it as ground truth. It contains:

product

Verkada's 8 product lines, positioning, and key differentiators (cloud-native, single platform, zero-appliance, AI at edge, NDAA-compliant)

icp

6 verticals (K-12, Healthcare, Retail, Manufacturing, Higher Ed, CRE) with key drivers and typical pain points

displacement_targets

8 incumbent vendors (Avigilon, Genetec, Milestone, Lenel, Hikvision, Dahua, March Networks, Brivo) with pain themes and Verkada counters

triggers

Detection signals with weights (0.0–1.0) mapped to discovery question templates. Free-form question generation is prohibited.

personas

6 buyer personas (IT Director, Facilities, CSO, VP Ops, LP Director, Superintendent) with care_about and skip_topics

disqualifiers

Flags that should kill an opportunity — prevents wasted SE time on unwinnable deals

Rule: Do not invent triggers, personas, or templates that aren't in this file. If you find a gap, surface it — don't fill it silently.

MEDDIC Framework

Every brief includes a MEDDIC qualification section synthesized from cross-subagent data. Each field includes value, evidence array, confidence level, gap analysis, and source quality rating.

M
Metrics — Quantified business impact Verkada can deliver (site consolidation, NVR elimination, NDAA compliance)
E
Economic Buyer — Mapped from leadership data + persona meddic_role mappings. Named individuals when available.
D
Decision Criteria — What the buyer evaluates on, derived from vertical drivers + pain hypotheses + NDAA exposure
D
Decision Process — How they buy: board approval (K-12), cooperative purchasing (gov), procurement dept (enterprise)
I
Identify Pain — Highest-confidence pain hypothesis that maps to a Verkada product capability
C
Champion — Named individual from champion signals enrichment. Highest champion_fit_score with meddic_role=champion.

The brief's interactive MEDDIC checklist lets SEs track which fields have been validated during discovery calls.

Verkada GTM Strategy

Each brief includes a tailored go-to-market strategy derived from the account's vertical, size, pain hypotheses, and competitive landscape.

Commands

The platform exposes slash commands for different research and prep workflows. Only /research fetches new data — all other commands read from cache.

Command Purpose
/research <company> Full pipeline: dispatch source clients, run subagents, synthesize, render HTML
/displace <company> Competitive displacement battle card from cached brief + persona
/vertical-lens <company> Vertical-specific angle overlay (K-12, healthcare, etc.)
/discovery-prep <company> <persona> Persona-tuned discovery questions from cached brief
/demo-plan <company> Recommended Command modules + 30-min demo script
/size <company> Rough product mix from sites/doors/cameras inputs
/followup <notes> Post-call email drafter from persona only

Architecture

The platform follows a strict layered architecture with clear boundaries between data collection, analysis, synthesis, and presentation. 

clients/          20 source clients (one per data source)
  └→ sources/     Raw JSON cache with retrieved_at timestamps

.claude/agents/   Subagent definitions (company-bg, tech-and-pain, hiring-signals)
  └→ Synthesizer  Main + deals synthesizer (Opus)

persona/          verkada-se.yml rule engine
  └→ Used by      Synthesizers + module commands

templates/        Jinja2 HTML templates
render/           JSON → HTML interpolation
briefs/           Final deliverables (.json + .html)
chat_server.py    FastAPI server (homepage, chat, API, status)

Caching

All raw source data cached to sources/<company>/. Only /research refreshes stale cache. Other commands fail loudly if cache is missing.

Brief Contract

The brief JSON in briefs/<company>-<date>.json is the contract between /research and all module commands. Schema must not be broken.

Confidence Pipeline

Confidence levels propagate through the pipeline — never upgraded. high = sourced, medium = inferred with evidence, inference = hypothesis.
